PhxModel

Co-simulate a PHX scene from Simulink®

Simulink block  |  Library: PhxLibrary.slx

Description

The PhxModel block runs a PHX physics scene inside a Simulink model. The scene is built exactly as it is from MATLAB — a script or MAT file that creates phx.Body objects and their children — and the block wraps it in a phx.Simulation that is stepped once per Simulink sample time.

PHX keeps its code-first identity: there is a single co-simulation block rather than a palette of physical primitives. You author the scene as ordinary phx.* code and use the block only to exchange signals with the rest of the Simulink model. Each input and output port is bound to a public property of a named scene object, so any quantity that is readable or writable from MATLAB can be wired into Simulink.

The block is backed by the Level-2 MATLAB S-function phx/PhxModel.m and the support class phx.simulink.BlockBackend. It is a viewing device: it can optionally open an interactive phx.extra.Viewer and even emit a rendered camera image as a signal.

Ports

Ports are not fixed: you declare them as text in the block mask. Each line is a parameter reference naming one scene object and one of its properties.

Reference syntax

ObjectName.Property ObjectName.Property(indices)

ObjectName is the Name of a phx.base.Object in the scene; Property is any of its public properties. The bracket-less form binds the whole property and the signal width is resolved automatically from class metadata (for example Body1.Position is a 3-element signal). The bracketed form selects linear indices, for example Body1.Quaternion(1:4) or Body1.TotalForce(3).

Separate several references with new lines or commas. Input references must name a writable property; read-only properties may be used only as outputs.

Input ports write the incoming signal into the referenced property at every step (for example driving Position of a kinematic body, or a Thruster force). Output ports read the referenced property after the step (for example Position, Quaternion or Energy). References are validated when you edit the mask (for MAT-file sources) and again when the simulation starts; unknown objects or properties raise a phx:PhxModel:* error.

Method ports. References such as Body1.applyForce are recognised but not yet supported and raise phx:PhxModel:methodPortsUnsupported. Drive bodies through writable properties instead.

Parameters

ParameterType / DefaultDescription
Sourcechar/string Scene source: the name of a model script (.m) that builds and returns the bodies, or a .mat file holding the scene. Loaded once when the simulation starts.
InputRefstext list One parameter reference per input port, e.g. Plate.Orientation(1:2). Separate with new lines or commas.
OutputRefstext list One parameter reference per output port, e.g. Ball.Position.
Substeps1 Number of engine substeps per Simulink sample time; the engine step is SampleTime/Substeps. More substeps stabilise stiff constraint networks.
SampleTime[] Block sample time, in seconds. Each sample time advances the scene by that interval.
ShowViewerfalse Open an interactive phx.extra.Viewer figure that renders the scene while the model runs.
RenderEachSteptrue When the viewer is on, redraw on every substep (true) or only once per sample time (false).
DefaultCameraPosition[-10 -10 10] Initial viewer camera position, in world coordinates.
DefaultCameraTarget[0 0 0] Initial viewer camera target, in world coordinates.
CameraResolution[] Optional rendered-image output. When set to [H W] (and the viewer is on), an extra uint8 output port emits the rendered scene as an [H W 3] image signal.

An Edit ports… button in the mask opens a picker (phx.simulink.portEditor) that lists the objects and sizeable properties of the scene so you can add input and output references without typing them by hand.

Building a model

1. Write a scene source. A model script names the objects it wants to expose:

% model_pendulum.m — build a scene and name the objects to expose
function bodies = model_pendulum
    ax = cla(clf); view(3); axis equal; grid on; camlight headlight;

    anchor = phx.Body(ax, "Type", "static", "Name", "Anchor");
    bob    = phx.Body(ax, "Position", [1 0 0], "Name", "Bob");
    phx.RevoluteJoint(anchor, bob, "Point", [0 0 0], "Axis", [0 1 0]);

    bodies = [anchor bob];
end

2. Drop a PhxModel block from PhxLibrary.slx into your model.

3. Set Source to model_pendulum, choose a SampleTime, and declare ports, for example output Bob.Position and input Bob.TotalForce.

4. Wire the ports to the rest of the model and run. Turn ShowViewer on to watch the scene.

Examples

Read a body's position into Simulink

With Source set to a scene that contains a body named Ball, declare a single output reference to stream its world position:

% OutputRefs (one line)
Ball.Position

The output port carries a 3-element signal; the width is taken from phx.Body.Position automatically.

Drive a kinematic body and read several signals

% InputRefs
Plate.Orientation(1:2)

% OutputRefs
Ball.Position
Ball.LinearVelocity

For a kinematically pose-driven body, keep Substeps at 1 and use a smaller sample time — with several substeps the whole pose increment is applied on the first substep, so the body appears to move N× faster between samples.

Ready-made co-simulation models live in examples/: phxex_sim_stand.slx, phxex_sim_ballplate2.slx and phxex_sim_sorter.slx, each paired with a model_*.m scene builder.

Stability. Tight constraint networks need a small engine step (about dt ≤ 5 ms). Reach this either with a small SampleTime or by raising Substeps so that SampleTime/Substeps stays small.
