PhxAction

Run MATLAB code against a running PHX scene

Simulink block  |  Library: PhxLibrary.slx

Description

The PhxAction block executes a snippet of user MATLAB code against the live scene of a PhxModel block during simulation. It complements the property-driven ports of PhxModel: where PhxModel streams signals to and from public properties every step, PhxAction lets you run arbitrary code on demand — to make structural changes such as adding or removing bodies, or to make an occasional method call — that does not fit the fixed, per-step port interface.

The code runs only on a rising edge of the block's trigger input, so actions are event-driven rather than executed every step. Between triggers the block simply holds its last outputs.

PhxAction binds to the scene-defining PhxModel block in the same model: it shares that block's phx.Simulation and viewer axes, so your code operates on exactly the objects PhxModel is stepping. The block is the Level-2 MATLAB S-function phx/PhxAction.m, backed by phx.simulink.ActionBackend.

The action code

The Code parameter is ordinary MATLAB code that runs with the following variables in scope. It reads the scene through sim, may keep data across triggers in state, and assigns the output ports by writing output1…outputM.

VariableTypeDescription
simphx.Simulation The running simulation of the bound PhxModel block. Use it to read the scene or change its structure, e.g. sim.addObjects(…) or delete(…).
axaxes | [] The viewer axes of the bound block, or [] when its viewer is off. Pass it as the parent when creating bodies that should be drawn.
tdouble Current simulation time, in seconds.
input1…inputNdouble The data input ports, in order (see NumInputs). Sizes are inherited from the connected signals.
statestruct Persistent user state, preserved across triggers. Read and write freely; initially an empty struct.
output1…outputMdouble Assign these to set the data output ports (see OutputSizes). Any output left unassigned defaults to 0.
Structural changes. Adding or removing objects rebuilds the simulation pipelines automatically (see phx.Simulation). Trigger such actions sparingly — on events rather than every step — and keep stiff constraint networks in mind (a small engine step, about dt ≤ 5 ms, remains necessary for stability).

Parameters

ParameterType / DefaultDescription
Codechar The MATLAB code to run on each rising trigger edge, using the variables above.
NumInputs0 Number of data input ports, in addition to the trigger. They appear as input1…inputN and inherit their sizes.
OutputSizes{} Cell array of output-port dimensions, one entry per output, e.g. {1, 3, [2 4]} for a scalar, a 3-vector and a 2×4 matrix. The number of outputs is numel(OutputSizes).
MainBlockSID The block to bind to, given as a Simulink block SID. Leave empty to auto-bind to the single PhxModel block in the model.

Ports

Input 1 is the trigger: the code runs whenever this signal crosses from ≤ 0 to > 0. Inputs 2…1+NumInputs are data inputs, exposed in the code as input1…inputN with inherited sizes. Outputs 1…M carry the values assigned to output1…outputM, sized per OutputSizes; each holds its last value between triggers.

Examples

Spawn a body on a trigger

With NumInputs = 0 and OutputSizes = {1}, drop a new box into the scene each time the trigger fires and report how many have been dropped:

% Code
if ~isfield(state, "n"), state.n = 0; end
state.n = state.n + 1;

b = phx.Body(ax, "Position", [0 0 5], "Name", "Drop" + state.n);
sim.addObjects(b);                 % rebuilds the pipelines

output1 = state.n;                 % number of bodies dropped so far

Remove a body using an input

With NumInputs = 1, delete a named body when triggered; the name index arrives on input1:

% Code
target = sim.findBy("Name", "Drop" + input1);
if ~isempty(target)
    delete(target{1});             % also rebuilds the pipelines
end
Binding. Place a PhxModel block in the same model. PhxAction resolves its target on the first step: it auto-binds to the single PhxModel block, or to the block named by MainBlock. With no PhxModel block present the simulation errors with phx:PhxAction:noMainBlock.
